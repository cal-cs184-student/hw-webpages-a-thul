Part 1: Ray Generation and Scene Intersection

Ray Generation and Primitive Intersection Pipeline

The ray generation works by taking a pixel coordinate (between 0 and 1 in each dimension), and mapping it onto a point in the sensor plane in the camera space (using its vertical and horizontal span, as determined by FOV.) Then, I used the c2w matrix to convert this sensor plane point to the actual world space. Then, the normalized vector in this direction is returned. The triangle intersection algorithm (Möller-Trumbore) is talked about in more detail below, and the sphere intersection works by solving the quadratic ray equation (if the determinant is positive) that was covered in lecture.

Triangle Intersection Algorithm

I implemented the Möller-Trumbore algorithm that was briefly mentioned in lecture. It functions by relying on the fact that, if the intersection point p is inside the triangle, then it can be expressed as b1p1 + b2p2 + (1-b1-b2)p3. We also know that p = O+tD for the ray, and that b1 + b2 <= 1, b1, b2 >= 0. The system is then solved (3 components, 3 unknowns b1 b2 and t) via Cramer's rule (I referred to the wikipedia page for the details, the cross products in the code function as determinants, and the dot products are part of solving the linear system via Cramer's).

Part 2: Bounding Volume Hierarchy

BVH Construction Algorithm

The BVH construction algorithm starts by computing the bounding box of all primitives within the provided ray range. Then, as given in the hint, it finds the longest axis (x, y, or z) of this bounding box, and computes the average corresponding centroid along this axis (e.g. average centroid x coordinate) of all primitives to determine where to split. Then, the primitives are split depending on whether the centroid is to the left or right of this split plane. The left and right child nodes are then created and recursively constructed. In the case that all primitives have the same centroid coordinate on that axis, it splits the list in half arbitrarily.

BVH Speedup Analysis

With an M3 Macbook Pro, with maxplanck.dae, running without BVH acceleration took 17.71s, while running with BVH acceleration took 0.03s. On cow.dae, running without BVH acceleration took 1.75s, while running with BVH acceleration took 0.03s. We can see that implementing BVH acceleration improved the performance of the renderer by orders of magnitude, down to the point where the actual path tracer is no longer the bottleneck, and everything is reduced to near the runtime of the overhead cost (0.03s regardless of complexity). We can see that we have successfully decreased the scaling time complexity of the problem to O(log N) from O(N) (on average).

Part 3: Direct Illumination

Direct Lighting Implementations

For uniform hemisphere sampling, we sample directions uniformly over the hemisphere at each hit surface's normal. If rays along these directions hit something, grab that object's emission, weight that ray by the bsdf * emission * cos(theta) * 2pi, and average to get the radiance value of that surface. For importance sampling, per surface, we only sample directions toward light surfaces. Then, we see if rays along these directions are blocked. If not, then we weight it by bsdf * emission * cos(theta) / pdf (where pdf is the probability function of the light returning this direction, geometry dependent). We average it per light, and then sum over lights to get the radiance value of that surface.

Hemisphere vs. Light Sampling Comparison

The uniform hemisphere sampling images are far more grainy than the lighting sampling ones. This lines up with my understanding, as the number of samples between the two methods is the same, but the lighting sampling has a far higher chance of each sample contributing to the radiance (as it is in the direction of a light source). On the other hand, many of the uniform sampling rays do not hit any lights, so they don't contribute any radiance and are wasted. This results in higher variance/grain in the image.

Part 4: Global Illumination

Indirect Lighting Implementation

The indirect lighting function works by computing the direct lighting at a given point, then using the BSDF to sample a new direction of reflection (total radiance weighted down by bsdf*cos(theta)/pdf), and then recursively calling it if it hits another surface. This is done to a finite depth, before terminating. However, this implementation is naive, as it cuts off light that would have bounced off successive surfaces, so I implemented a russian roulette termination that terminate with probability 0.35 (after the first bounce; also I made it continue with probability 0.65 which is equivalent), at each sucessive bounce.

Part 5: Adaptive Sampling

Adaptive Sampling Explanation

For certain areas of the image (surfaces), the radiance values converges to close to the true radiance value very quickly (with fewer samples), while other areas take higher numbers of samples to converge. To account for this and not waste samples on surfaces that converge quickly, we use adaptive sampling. I implemented the adaptive sampling method as per the spec, where we sample in batches, calculate the mean and variance of radiance over the total number of samples. We then calculate the radius of the 95% confidence interval, and if this radius is lower than our defined tolerance, then we terminate (as the true value is within tolerance with a 95% probability).
